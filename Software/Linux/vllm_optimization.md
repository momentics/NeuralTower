# Оптимизация vLLM и подсистемы памяти для NeuralTower

Для эффективного использования четырех ускорителей Tesla V100 и преодоления физического лимита видеопамяти в 128 ГБ, программный стек системы требует глубокой настройки на уровне ядра Linux и параметров движка vLLM. Основная задача - обеспечить стабильную работу механизма SWAP на NVMe для расширения контекста и минимизировать задержки при обмене данными между GPU.

## 1. Подготовка операционной системы

Перед настройкой vLLM система должна пройти аппаратную инициализацию: BIOS, PCIe-адресация, видимость всех GPU и проверка топологии. Порядок подготовки описан в [system_setup.md](./system_setup.md), настройки BIOS - в [../../Hardware/BIOS/bios_settings.md](../../Hardware/BIOS/bios_settings.md).

Для vLLM критично, чтобы `nvidia-smi topo -m` показывал NVLink внутри пар GPU 0/1 и GPU 2/3, а межмезонинный обмен шел через ожидаемые PCIe/PLX-связи.

## 2. Ситуация с vLLM и V100

Начиная с vLLM 0.20.0 (27 апреля 2026), официальная поддержка sm_70 полностью убрана из build-системы. Выпуск переключился на CUDA 13.0 по умолчанию, PyTorch 2.11+, и списки целевых архитектур очищены от Volta. Установка vLLM 0.20+ на V100 завершается ошибкой `CUDA error: no kernel image is available for execution on the device`. Документированных обходных путей в официальном репозитории нет.

Это оставляет два рабочих варианта:
1. **1Cat-vLLM** — форк с восстановленной поддержкой V100, FlashAttention-2 и AWQ 4-bit
2. **Официальный vLLM ≤0.18.x** — последняя версия с нативной поддержкой sm_70

### 2.1. Вариант 1: 1Cat-vLLM

Проект [1Cat-vLLM](https://github.com/1CatAI/1Cat-vLLM) — это форк vLLM, специально адаптированный под V100. Он добавляет:

- **FLASH_ATTN_V100** — портированное FlashAttention-2 для sm_70
- **AWQ 4-bit** — квантизация через ядра lmdeploy TurboMind, работающая на Volta
- Валидацию на 4× V100 32 ГБ с моделями Qwen3.5/Qwen3.6
- Готовые wheel-пакеты для CUDA 12.8 + Python 3.12

#### 2.1.1. Установка 1Cat-vLLM

```bash
# CUDA 12.8 обязателен для этого форка
export CUDA_HOME=/usr/local/cuda-12.8
export PATH=$CUDA_HOME/bin:$PATH
export LD_LIBRARY_PATH=$CUDA_HOME/lib64:${LD_LIBRARY_PATH:-}

# Установка готовых wheel
python -m pip install --prefer-binary --no-cache-dir \
  --extra-index-url https://download.pytorch.org/whl/cu128 \
  "https://github.com/1CatAI/1Cat-vLLM/releases/download/v1.0.0/flash_attn_v100-1.0.0-cp312-cp312-linux_x86_64.whl" \
  "https://github.com/1CatAI/1Cat-vLLM/releases/download/v1.0.0/vllm-1.0.0-cp312-cp312-linux_x86_64.whl"
```

Запуск из директории вне исходного дерева (иначе Python подхватит локальный `vllm/` без скомпилированных CUDA-расширений):

```bash
cd ~
NCCL_P2P_LEVEL=NVL \
NCCL_IB_DISABLE=1 \
NCCL_NET_PLUGIN=none \
NCCL_DEBUG=WARN \
python -m vllm.entrypoints.openai.api_server \
  --model /path/to/Qwen3.5-27B-AWQ \
  --attention-backend FLASH_ATTN_V100 \
  --tensor-parallel-size 2 \
  --pipeline-parallel-size 2 \
  --gpu-memory-utilization 0.88 \
  --max-model-len 262144 \
  --max-num-seqs 1 \
  --max-num-batched-tokens 16384
```

Tensor Parallelism (TP) требует AllReduce-операций на каждом слое модели. При TP=4 все 4 GPU участвуют в каждом AllReduce, что означает обязательный обмен данными между мезонинами через PCIe 3.0 x16 (~16 ГБ/с). NVLink внутри мезонина обеспечивает ~300 ГБ/с — почти 20× быстрее.

Для Qwen3.6-27B (64 слоя, hidden_size=5120):
- **TP=4**: ~128 AllReduce/forward pass, каждый требует кросс-мезонинного трафика → bottleneck на PCIe
- **TP=2 + PP=2**: AllReduce только внутри NVLink-пар (GPU 0↔1, GPU 2↔3). Между мезонинами передаются только активации между PP-стадиями — один раз на запрос, а не на каждый слой

vLLM назначает GPU в TP-группы последовательно: GPU 0,1 → TP group 0 (Мезонин 1), GPU 2,3 → TP group 1 (Мезонин 2). PP-коммуникация идёт между стадиями через PCIe, но объём данных на порядки меньше, чем при TP=4.

### 2.2. Вариант 2: официальный vLLM ≤0.18.x

Если 1Cat-vLLM не подходит, можно использовать последний официальный vLLM с поддержкой sm_70:

```bash
pip install vllm==0.18.0

export VLLM_ATTENTION_BACKEND=TRITON_ATTN
export VLLM_TENSORRT_LLM_TIMEOUT=600
export NCCL_P2P_LEVEL=NVL
export NCCL_IB_DISABLE=1
export NCCL_NET_PLUGIN=none

python -m vllm.entrypoints.openai.api_server \
  --model <name> \
  --tensor-parallel-size 2 \
  --pipeline-parallel-size 2 \
  --dtype float16 \
  --gpu-memory-utilization 0.95 \
  --swap-space 64 \
  --max-model-len 32768
```

Triton медленнее FA2, но работает стабильно. Первый запуск модели компилирует JIT-ядра 5-10 минут.

`--pipeline-parallel-size 2` разделяет 64 слоя модели на две PP-стадии по 32 слоя каждая. Мезонин 1 (GPU 0,1) executes стадию 0, Мезонин 2 (GPU 2,3) executes стадию 1. Активации передаются от стадии 0 к стадии 1 через PCIe, но объём данных (~5 МБ/токен при batch=32) на порядки меньше, чем AllReduce трафик при TP=4 (~10.5 МБ/токен на каждый из 64 слоёв).

## 3. Ограничения Volta (sm_70) и обходные пути

V100 — архитектура 2017 года. CUDA 13.x помечает sm_70 как legacy, и современные фреймворки по умолчанию ориентируются на Ampere/Hopper. Для стабильной работы с V100 в NeuralTower используется CUDA 12.8, которая сохраняет полную поддержку sm_70.

### 3.1. Точность и типы данных

V100 поддерживает FP16 и BF16 (эмулированный, работает медленнее). FP8 и MXFP4 недоступны — это требует sm_80+.

```bash
# Фиксируем FP16, иначе vLLM может попробовать FP8 и упасть
--dtype float16
```

### 3.2. Застревание при инициализации

В версиях vLLM 0.16.x есть баг: некоторые модели «тихо зависают» при старте на V100 — deadlock в инициализации памяти. Обходной путь:

```bash
--max-model-len 32768
```

Явное ограничение длины модели предотвращает попытку vLLM аллоцировать больше памяти, чем доступно, и обходит deadlock.

### 3.3. Квантизация для больших моделей

Для моделей класса 70B, которые не влезают в 128 ГБ даже с SWAP, используем 4-битную квантизацию. С 1Cat-vLLM это AWQ:

```bash
--quantization awq
```

Без 1Cat-vLLM можно использовать bitsandbytes, но это медленнее.

## 4. Настройка механизма SWAP на NVMe

Ключевой особенностью NeuralTower является использование vLLM с активированным вытеснением кэша на быстрые NVMe-накопители. Это позволяет обрабатывать запросы с контекстом, превышающим доступный объем HBM2, за счет динамической выгрузки блоков памяти.

Для минимизации задержек в этом процессе необходимо:
1. Выделить под swap-пространство отдельный NVMe-накопитель с интерфейсом PCIe 3.0, подключенный напрямую к линиям процессора, а не через чипсет.
2. Использовать файловую систему XFS или вовсе сырой раздел (raw partition) для уменьшения накладных расходов на журналирование.

## 5. Мониторинг и управление

Поскольку система потребляет до 1.4 кВт, критически важно отслеживать состояние памяти и температур в реальном времени. В состав NeuralTower входит кастомный скрипт мониторинга, который через `nvidia-ml-py` агрегирует данные со всех четырех GPU.

 Особое внимание уделяется параметру `Memory Fragmentation`. При активном использовании SWAP на NVMe важно следить, чтобы скорость обмена данными по шине PCIe не становилась "бутылочным горлышком". Для этого в системе настраиваются параметры `Current PCIe Link Speed` и `Link Width`. Каждый SlimSAS-кабель физически несёт x8 Gen3, но на стороне мезонина два кабеля от одного GPU снова объединяются в полноценный x16-поток для кристалла V100. Бифуркация на уровне материнской платы не требуется — коммутатор PLX видит одно устройство x16 на каждый слот.

## 6. Топология PLX-коммутаторов и балансировка PCIe-трафика

Материнская плата ASUS X99-E WS оснащена двумя коммутаторами **PLX PEX8747**, каждый из которых забирает по 16 линий PCIe 3.0 от процессора Intel Xeon и размножает их на физические слоты. Понимание этой топологии критично для корректной работы GPUDirect P2P и предотвращения перегрузки одного из мостов.

### 6.1. Архитектурное распределение слотов

| Чип PLX | Управляемые слоты | Назначение в NeuralTower |
|---------|-------------------|--------------------------|
| **PLX1 (верхний)** | PCIEX16_1, PCIEX16_2, PCIEX16_3, PCIEX16_4 | GPU 0 (Слот 1) + GPU 1 (Слот 3) → Мезонин 1 |
| **PLX2 (нижний)** | PCIEX16_5, PCIEX16_6, PCIEX16_7 (+ USB/SATA) | GPU 2 (Слот 5) + GPU 3 (Слот 7) → Мезонин 2 |

### 6.2. Почему именно слоты 1, 3, 5, 7?

Коммутаторы PEX8747 разведены так, что при occupation нечётных слотов с пропуском чётных каждый адаптер получает **честные физические x16 линий** от моста PLX:
- Слот 1 занят, Слот 2 пуст → Слот 1 работает в режиме x16 Gen3
- Слот 3 занят, Слот 4 пуст → Слот 3 работает в режиме x16 Gen3
- Аналогично для пар 5/6 и 7/8

**Промежуточные слоты (2, 4, 6) обязательно остаются пустыми.**

### 6.3. Оптимизация межмезонинного обмена (GPUDirect P2P)

1. **Локализация Tensor Parallelism внутри мезонинов (NVLink):** При `--tensor-parallel-size 2` все AllReduce-операции (синхронизация весов на каждом из 64 слоёв) идут внутри пар GPU 0↔1 и GPU 2↔3 напрямую через NVLink (~300 ГБ/с), вообще не нагружая материнскую плату и чипы PLX.

2. **Pipeline Parallelism между мезонинами (PCIe):** При `--pipeline-parallel-size 2` модель разделена на две стадии по 32 слоя. Активации передаются от Мезонина 1 к Мезонину 2 через PCIe 3.0 x16 (~16 ГБ/с). Для batch=32 и hidden_size=5120 объём передачи составляет ~5 МБ/токен, что занимает ~0.3 мс — пренебрежимо мало по сравнению с вычислениями на 32 слоях.

3. **Локальная коммутация внутри PLX:** Когда GPU 0 общается с GPU 1 по PCIe (служебные команды, инициализация), этот трафик коммутируется внутри кристалла PLX1 на полной скорости x16 Gen3, не поднимаясь к процессору. Аналогично для GPU 2↔3 внутри PLX2.

4. **Межчиповый баланс:** PP-трафик между мезонинами асимметричен (стадия 0 → стадия 1 в forward, обратно в backward), но объём данных мал. Нагрузка на PLX1 и PLX2 распределена равномерно, так как каждый мезонин отправляет и получает данные одинакового объёма.

#### 6.3.1. NCCL-оптимизация для топологии NeuralTower

Переменные окружения критически важны для корректного маршрутизирования трафика:

| Переменная | Значение | Эффект |
|-----------|---------|--------|
| `NCCL_P2P_LEVEL=NVL` | Принудительно | NCCL использует P2P-обмен только через NVLink, а не через PCIe |
| `NCCL_IB_DISABLE=1` | Принудительно | Отключает поиск InfiniBand (нет в системе) |
| `NCCL_NET_PLUGIN=none` | Принудительно | Отключает сетевой плагин NCCL (не нужен для single-node) |
| `NCCL_DEBUG=WARN` | Рекомендовано | Логирование только предупреждений; переключите на `INFO` для отладки |

**Верификация топологии:** После запуска выполните `nvidia-smi topo -m` и убедитесь, что GPU 0↔1 и GPU 2↔3 помечены как `NV#` (NVLink), а кросс-мезонинные связи — как `PIX` (PCIe).

## 7. Распределение памяти для Qwen3.6-27B

При TP=2 + PP=2 модель делится следующим образом:

| Компонент | Размер | На GPU |
|-----------|--------|--------|
| Веса модели (FP16) | ~54 ГБ | ~13.5 ГБ (TP=2) |
| KV cache (batch=32, 4K ctx) | ~8 ГБ | ~2 ГБ |
| Активации (32 слоя/стадия PP) | ~4 ГБ | ~2 ГБ |
| **Итого (без запаса)** | | **~17.5 ГБ** |
| Доступно (32 ГБ × 0.88) | | **~28.1 ГБ** |
| **Свободно для KV cache** | | **~10.6 ГБ** |

При `--gpu-memory-utilization 0.88` каждый GPU резервирует ~28 ГБ. Это оставляет ~10 ГБ для KV cache, что достаточно для batch size 32+ при контексте 4K. Для longer context уменьшайте `--gpu-memory-utilization` или используйте `--swap-space` для вытеснения на NVMe.

## 8. Оптимизация пропускной способности через NVLink

Внутри мезонинов карты соединены парами через NVLink. Программная среда настраивается таким образом, чтобы vLLM в первую очередь передавал данные между "соседними" GPU внутри мезонина, и только при необходимости задействовал более медленную шину PCIe для межмезонинного обмена. Это достигается путем правильной нумерации устройств в переменной окружения `CUDA_VISIBLE_DEVICES`, соответствующей физической топологии кабелей SlimSAS.

vLLM назначает GPU в TP-группы последовательно: первые `--tensor-parallel-size` GPU формируют TP group 0 (PP stage 0), следующие — TP group 1 (PP stage 1). Поэтому порядок `0,1,2,3` гарантирует, что GPU 0 и 1 (Мезонин 1, связаны NVLink) попадут в одну TP-группу, а GPU 2 и 3 (Мезонин 2, связаны NVLink) — в другую.
