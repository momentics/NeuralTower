# Оптимизация vLLM и подсистемы памяти для NeuralTower

Для эффективного использования четырех ускорителей Tesla V100 и преодоления физического лимита видеопамяти в 128 ГБ, программный стек системы требует глубокой настройки на уровне ядра Linux и параметров движка vLLM. Основная задача - обеспечить стабильную работу механизма SWAP на NVMe для расширения контекста и минимизировать задержки при обмене данными между GPU.

## Подготовка операционной системы

Стандартные настройки распределения памяти в Linux не рассчитаны на одновременную работу с четырьмя картами по 32 ГБ каждая через мосты PLX. Для корректной инициализации всего адресного пространства в параметрах загрузки ядра (GRUB) необходимо активировать поддержку 64-битного BAR.

В файле `/etc/default/grub` в строку `GRUB_CMDLINE_LINUX_DEFAULT` вносятся следующие параметры:
`pci=realloc,assign-busses pci=hp_pcie_bus_max`
Это гарантирует, что система не "потеряет" видеокарты при нехватке адресов, зарезервированных под другие устройства PCIe.

## Настройка vLLM и механизма SWAP на NVMe

Ключевой особенностью NeuralTower является использование vLLM с активированным вытеснением кэша на быстрые NVMe-накопители. Это позволяет обрабатывать запросы с контекстом, превышающим доступный объем HBM2, за счет динамической выгрузки блоков памяти.

Для минимизации задержек в этом процессе необходимо:
1. Выделить под swap-пространство отдельный NVMe-накопитель с интерфейсом PCIe 4.0, подключенный напрямую к линиям процессора, а не через чипсет.
2. Использовать файловую систему XFS или вовсе сырой раздел (raw partition) для уменьшения накладных расходов на журналирование.

При запуске vLLM используются специфические флаги:
`--gpu-memory-utilization 0.95` - резервирование 95% памяти под веса и кэш.
`--swap-space 32` - выделение 32 ГБ системной памяти (или NVMe) под нужды свопа блоков.
`--tensor-parallel-size 4` - распараллеливание модели на все четыре ускорителя.

## Мониторинг и управление подсистемой мониторинга

Поскольку система потребляет до 1.4 кВт, критически важно отслеживать состояние памяти и температур в реальном времени. В состав NeuralTower входит кастомный скрипт мониторинга, который через `nvidia-ml-py` агрегирует данные со всех четырех GPU. 

Особое внимание уделяется параметру `Memory Fragmentation`. При активном использовании SWAP на NVMe важно следить, чтобы скорость обмена данными по шине PCIe не становилась "бутылочным горлышком". Для этого в системе настраиваются параметры `Current PCIe Link Speed` и `Link Width`, которые должны жестко удерживаться в режиме x8 Gen3 для каждого линка SlimSAS.

## Оптимизация пропускной способности через NVLink

Внутри мезонинов карты соединены парами через NVLink. Программная среда настраивается таким образом, чтобы vLLM в первую очередь передавал данные между "соседними" GPU внутри мезонина, и только при необходимости задействовал более медленную шину PCIe для межмезонинного обмена. Это достигается путем правильной нумерации устройств в переменной окружения `CUDA_VISIBLE_DEVICES`, соответствующей физической топологии кабелей SlimSAS.
