# Оптимизация vLLM и подсистемы памяти для NeuralTower

Для эффективного использования четырех ускорителей Tesla V100 и преодоления физического лимита видеопамяти в 128 ГБ, программный стек системы требует глубокой настройки на уровне ядра Linux и параметров движка vLLM. Основная задача - обеспечить стабильную работу механизма SWAP на NVMe для расширения контекста и минимизировать задержки при обмене данными между GPU.

## Подготовка операционной системы

Стандартные настройки распределения памяти в Linux не рассчитаны на одновременную работу с четырьмя картами по 32 ГБ каждая через мосты PLX. Для корректной инициализации всего адресного пространства в параметрах загрузки ядра (GRUB) необходимо активировать поддержку 64-битного BAR.

В файле `/etc/default/grub` в строку `GRUB_CMDLINE_LINUX_DEFAULT` вносятся следующие параметры:
`pci=realloc,assign-busses pci=hp_pcie_bus_max`
Это гарантирует, что система не "потеряет" видеокарты при нехватке адресов, зарезервированных под другие устройства PCIe.

## Ситуация с vLLM и V100

Начиная с vLLM 0.20.0 (27 апреля 2026), официальная поддержка sm_70 полностью убрана из build-системы. Выпуск переключился на CUDA 13.0 по умолчанию, PyTorch 2.11+, и списки целевых архитектур очищены от Volta. Установка vLLM 0.20+ на V100 завершается ошибкой `CUDA error: no kernel image is available for execution on the device`. Документированных обходных путей в официальном репозитории нет.

Это оставляет два рабочих варианта:
1. **1Cat-vLLM** — форк с восстановленной поддержкой V100, FlashAttention-2 и AWQ 4-bit
2. **Официальный vLLM ≤0.18.x** — последняя версия с нативной поддержкой sm_70

### Вариант 1: 1Cat-vLLM

Проект [1Cat-vLLM](https://github.com/1CatAI/1Cat-vLLM) — это форк vLLM, специально адаптированный под V100. Он добавляет:

- **FLASH_ATTN_V100** — портированное FlashAttention-2 для sm_70
- **AWQ 4-bit** — квантизация через ядра lmdeploy TurboMind, работающая на Volta
- Валидацию на 4× V100 32 ГБ с моделями Qwen3.5/Qwen3.6
- Готовые wheel-пакеты для CUDA 12.8 + Python 3.12

#### Установка 1Cat-vLLM

```bash
# CUDA 12.8 обязателен для этого форка
export CUDA_HOME=/usr/local/cuda-12.8
export PATH=$CUDA_HOME/bin:$PATH
export LD_LIBRARY_PATH=$CUDA_HOME/lib64:${LD_LIBRARY_PATH:-}

# Установка готовых wheel
python -m pip install --prefer-binary --no-cache-dir \
  --extra-index-url https://download.pytorch.org/whl/cu128 \
  "https://github.com/1CatAI/1Cat-vLLM/releases/download/v1.0.0/flash_attn_v100-1.0.0-cp312-cp312-linux_x86_64.whl" \
  "https://github.com/1CatAI/1Cat-vLLM/releases/download/v1.0.0/vllm-1.0.0-cp312-cp312-linux_x86_64.whl"
```

Запуск из директории вне исходного дерева (иначе Python подхватит локальный `vllm/` без скомпилированных CUDA-расширений):

```bash
cd ~
python -m vllm.entrypoints.openai.api_server \
  --model /path/to/Qwen3.5-27B-AWQ \
  --attention-backend FLASH_ATTN_V100 \
  --tensor-parallel-size 4 \
  --gpu-memory-utilization 0.88 \
  --max-model-len 262144 \
  --max-num-seqs 1 \
  --max-num-batched-tokens 16384
```

#### Альтернатива: официальный vLLM ≤0.18.x

Если 1Cat-vLLM не подходит, можно использовать последний официальный vLLM с поддержкой sm_70:

```bash
pip install vllm==0.18.0

export VLLM_ATTENTION_BACKEND=TRITON_ATTN
export VLLM_TENSORRT_LLM_TIMEOUT=600

python -m vllm.entrypoints.openai.api_server \
  --model <name> \
  --tensor-parallel-size 4 \
  --dtype float16 \
  --gpu-memory-utilization 0.95 \
  --swap-space 64 \
  --max-model-len 32768
```

Triton медленнее FA2, но работает стабильно. Первый запуск модели компилирует JIT-ядра 5-10 минут.

## Ограничения Volta (sm_70) и обходные пути

V100 — архитектура 2017 года. CUDA 13.x помечает sm_70 как legacy, и современные фреймворки по умолчанию ориентируются на Ampere/Hopper. Для стабильной работы с V100 в NeuralTower используется CUDA 12.8, которая сохраняет полную поддержку sm_70.

### Точность и типы данных

V100 поддерживает FP16 и BF16 (эмулированный, работает медленнее). FP8 и MXFP4 недоступны — это требует sm_80+.

```bash
# Фиксируем FP16, иначе vLLM может попробовать FP8 и упасть
--dtype float16
```

### Застревание при инициализации

В версиях vLLM 0.16.x есть баг: некоторые модели «тихо зависают» при старте на V100 — deadlock в инициализации памяти. Обходной путь:

```bash
--max-model-len 32768
```

Явное ограничение длины модели предотвращает попытку vLLM аллоцировать больше памяти, чем доступно, и обходит deadlock.

### Квантизация для больших моделей

Для моделей класса 70B, которые не влезают в 128 ГБ даже с SWAP, используем 4-битную квантизацию. С 1Cat-vLLM это AWQ:

```bash
--quantization awq
```

Без 1Cat-vLLM можно использовать bitsandbytes, но это медленнее.

## Настройка механизма SWAP на NVMe

Ключевой особенностью NeuralTower является использование vLLM с активированным вытеснением кэша на быстрые NVMe-накопители. Это позволяет обрабатывать запросы с контекстом, превышающим доступный объем HBM2, за счет динамической выгрузки блоков памяти.

Для минимизации задержек в этом процессе необходимо:
1. Выделить под swap-пространство отдельный NVMe-накопитель с интерфейсом PCIe 4.0, подключенный напрямую к линиям процессора, а не через чипсет.
2. Использовать файловую систему XFS или вовсе сырой раздел (raw partition) для уменьшения накладных расходов на журналирование.

## Мониторинг и управление

Поскольку система потребляет до 1.4 кВт, критически важно отслеживать состояние памяти и температур в реальном времени. В состав NeuralTower входит кастомный скрипт мониторинга, который через `nvidia-ml-py` агрегирует данные со всех четырех GPU.

Особое внимание уделяется параметру `Memory Fragmentation`. При активном использовании SWAP на NVMe важно следить, чтобы скорость обмена данными по шине PCIe не становилась "бутылочным горлышком". Для этого в системе настраиваются параметры `Current PCIe Link Speed` и `Link Width`, которые должны жестко удерживаться в режиме x8 Gen3 для каждого линка SlimSAS.

## Оптимизация пропускной способности через NVLink

Внутри мезонинов карты соединены парами через NVLink. Программная среда настраивается таким образом, чтобы vLLM в первую очередь передавал данные между "соседними" GPU внутри мезонина, и только при необходимости задействовал более медленную шину PCIe для межмезонинного обмена. Это достигается путем правильной нумерации устройств в переменной окружения `CUDA_VISIBLE_DEVICES`, соответствующей физической топологии кабелей SlimSAS.
