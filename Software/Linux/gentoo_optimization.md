# Оптимизация среды NeuralTower на базе Gentoo Linux

Для реализации максимального потенциала связки Xeon E5-2699v4 и четырех ускорителей Tesla V100 выбрана операционная система Gentoo Linux. Это позволяет использовать преимущества новейших инструкций компилятора и специфических оптимизаций ядра для работы с тензорными вычислениями и высокоскоростным обменом данными по шине PCIe 3.0.

## 1. Настройка глобальных параметров сборки (make.conf)

Ключевым преимуществом Gentoo является возможность пересборки всей системы с флагами оптимизации под архитектуру Broadwell. Это критично для библиотек препроцессинга данных и драйверов, где важна минимальная задержка.

Основные параметры в `/etc/portage/make.conf`:
*   `COMMON_FLAGS="-O3 -march=broadwell -pipe -flto=auto"` - использование агрессивной оптимизации и межмодульной оптимизации времени линковки (LTO) для всех системных компонентов.
*   `USE="cuda nccl cudnn mpi tensorrt openmp opencl pcie-p2p -X -wayland -gui -dist-kernel"` - активация поддержки CUDA 12.8 и протоколов прямого обмена данными между GPU.
*   `PYTHON_TARGETS="python3_12"` - Python 3.12 требуется для совместимости с готовыми wheel-пакетами 1Cat-vLLM. При использовании Python 3.13 необходимо собирать vLLM из исходников (см. `world_build.md`, Конфигурация B).
*   `VIDEO_CARDS="nvidia"` - сборка драйверов актуальной ветки 5xx.x или новее.

## 2. Конфигурация ядра Linux 6.x+

Современные ядра Linux получили значительные улучшения в планировщике задач и подсистеме памяти, что критично для работы vLLM.

Необходимые опции при сборке:
*   `PCI_P2PDMA` - поддержка прямого доступа к памяти между устройствами на шине PCIe (Peer-to-Peer).
*   `NVME_TCP` и `NVME_MULTIPATH` - для оптимизации работы с NVMe-накопителем, используемым в качестве swap-пространства.
*   `HAVE_ARCH_HUGE_VMAP` - поддержка больших страниц памяти для ускорения работы CUDA-контекстов.
*   `NVIDIA_GDRV` - если используется модуль прямого доступа к памяти видеокарт.

## 3. Формирование программного стека ML

Стек машинного обучения в Gentoo требует согласованности следующих версий пакетов:

1. **Драйверы и тулкит:** `x11-drivers/nvidia-drivers` версии 580+ (стабильная поддержка sm_70) и `dev-util/nvidia-cuda-toolkit:12`.
2. **Библиотеки глубокого обучения:** `sci-libs/cudnn:9.x` и `sci-libs/nccl:2.20+` с поддержкой аппаратного ускорения внутри мезонинов.
3. **Контейнеризация:** `app-containers/docker` с поддержкой `nvidia-container-toolkit`, что позволяет запускать vLLM в изолированном окружении без потери производительности, достигнутой на уровне хост-системы.

## 4. Обновление @world

Для приведения системы в соответствие с выбранными флагами выполняется полная пересборка мира:
`emerge --ask --verbose --update --deep --newuse @world`

Такой подход гарантирует, что системные библиотеки (glibc, libstdc++), математические пакеты и драйверы будут работать как единый механизм, лишенный избыточного кода и использующий все 44 потока Xeon E5-2699v4 для управления очередями инференса. Это особенно важно при использовании механизма vLLM SWAP на NVMe, где задержки на уровне системных вызовов должны быть сведены к минимуму.
