# Настройка программной среды и оптимизация системы

NeuralTower требует специфической конфигурации на уровне BIOS и ядра Linux для обеспечения стабильной работы шины PCIe и управления 128 ГБ видеопамяти HBM2. Данный документ описывает этапы подготовки системы к запуску задач глубокого обучения и инференса.

## 1. Конфигурация BIOS ASUS X99-E WS

Для инициализации четырех ускорителей Tesla V100 и корректного распределения ресурсов необходимо выставить следующие параметры:

*   **Above 4G Decoding:** Enabled. Критически важная настройка для адресации больших объемов видеопамяти и работы P2P-интерконнекта.
*   **PCIe Speed:** Force Gen3. Установка фиксированного поколения шины предотвращает ошибки согласования на длинных линках SlimSAS.
*   **PCIe Bifurcation:** x8/x8 для всех слотов, где используются разветвители.
*   **VT-d / Virtualization:** Disabled (если не планируется использование проброса GPU в виртуальные машины) для минимизации задержек IOMMU.

## 2. Параметры загрузки ядра Linux

В файле `/etc/default/grub` необходимо модифицировать строку параметров загрузки для корректного распределения адресов PCIe мостами PLX:

`GRUB_CMDLINE_LINUX_DEFAULT="quiet splash pci=realloc,assign-busses pci=hp_pcie_bus_max"`

После внесения изменений необходимо обновить конфигурацию: `sudo update-grub`. Параметр `pci=realloc` принудительно пересчитывает ресурсы шины, что необходимо для создания сквозного окна адресации между мезонинами.

## 3. Активация и верификация GPUDirect P2P

Для объединения двух независимых мезонинов в единый вычислительный домен используется технология GPUDirect Peer-to-Peer. Она позволяет ускорителям обмениваться данными напрямую через коммутаторы PLX материнской платы, минуя системную оперативную память и центральный процессор.

Для работы данной технологии в Gentoo необходимо убедиться, что драйвер `nvidia-drivers` собран с поддержкой модуля `uvm`. Проверка статуса интерконнекта осуществляется командой `nvidia-smi topo -m`. В корректно настроенной системе NeuralTower матрица соединений должна отображать статус `NV#` для связей внутри мезонинов и `P2P` или `PIX` для межмезонинных соединений. Статус `SYS` указывает на некорректную настройку адресации и критическое падение производительности.

## 4. Развертывание среды для дообучения моделей

Для трансформации узла NeuralTower в станцию Fine-tuning используется стек на базе Unsloth и Axolotl. Это позволяет эффективно распределять градиенты между четырьмя ускорителями и минимизировать накладные расходы на хранение состояний оптимизатора.

### Оптимизация через Unsloth
Интеграция Unsloth позволяет сократить потребление видеопамяти на 40–50% при сохранении точности весов. В условиях 2026 года для архитектуры Broadwell и CUDA 13.x применяется ручная сборка ядер Triton, что обеспечивает прирост скорости вычислений на 25%. При обучении моделей класса Llama 3 (70B) рекомендуется использовать 4-битную квантизацию, что расширяет окно контекста до 32 тысяч токенов.

### Оркестрация через Axolotl
Для управления распределенным обучением применяется конфигуратор Axolotl с использованием стратегии FSDP (Fully Sharded Data Parallel). При настройке необходимо активировать параметр `sample_packing` для максимальной загрузки тензорных ядер V100 и исключения простоев вычислительного конвейера.

## 5. Движок инференса vLLM и механизм Swap-on-NVMe

Для работы с моделями, требующими контекста свыше физического объема памяти GPU, используется движок vLLM с активацией вытеснения кэша на NVMe-накопитель.

### Настройка Swap-пространства
Рекомендуется использовать выделенный NVMe-диск (например, Samsung 990 Pro или аналог с высоким ресурсом TBW). 
1. Создание раздела: `sudo mkswap /dev/nvmeXnX`
2. Установка приоритета: в `/etc/fstab` выставить `pri=100`.

### Параметры запуска vLLM
Для распределения модели на все четыре GPU и активации расширенного кэша:
`python -m vllm.entrypoints.openai.api_server --model <name> --tensor-parallel-size 4 --gpu-memory-utilization 0.95 --swap-space 64`

## 6. Мониторинг и тепловой контроль

Мониторинг температур VRM и GPU является обязательным условием стабильности. При дообучении моделей нагрузка на цепи питания возрастает на 30% относительно режима инференса. Критический порог температуры для памяти HBM2 составляет 75°C. Аналитический стек Prometheus и Grafana настроен на частоту опроса 5 секунд для мгновенной фиксации тепловых аномалий.
