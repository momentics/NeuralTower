# Порядок подготовки программной среды

Этот документ задает маршрут от включения железа до первого запуска модели. Подробные параметры BIOS, кабелей, Gentoo и vLLM вынесены в отдельные файлы; здесь оставлены только зависимости между шагами.

## 1. Аппаратные предусловия

Перед установкой программного стека должны быть выполнены проверки:

- BIOS настроен по [../../Hardware/BIOS/bios_settings.md](../../Hardware/BIOS/bios_settings.md): `Above 4G Decoding`, принудительный PCIe Gen3, UEFI-режим.
- Питание и земля проверены по [../../Electrical/Wiring/power_distribution.md](../../Electrical/Wiring/power_distribution.md) и [../../Electrical/Wiring/grounding_guide.md](../../Electrical/Wiring/grounding_guide.md).
- SlimSAS-кабели промаркированы по [../../Electrical/Pinouts/slimsas_mapping.md](../../Electrical/Pinouts/slimsas_mapping.md).
- После POST все GPU видны в `nvidia-smi`.

Если `nvidia-smi topo -m` показывает `SYS` там, где ожидается локальная связь через PLX/NVLink, программную настройку vLLM начинать рано: сначала надо исправить BIOS, кабели или адресацию PCIe.

## 2. Linux и драйверы

Для корректного распределения адресного пространства PCIe в параметрах загрузки ядра используется:

```bash
GRUB_CMDLINE_LINUX_DEFAULT="quiet pci=realloc,assign-busses pci=hp_pcie_bus_max"
```

После изменения конфигурации GRUB обновите загрузчик и выполните холодный старт. Детали Gentoo, флаги сборки, параметры ядра и `@world`-пересборка описаны в [gentoo_optimization.md](./gentoo_optimization.md).

Минимальные признаки готовности ОС:

- загружен NVIDIA-драйвер с `uvm`;
- все четыре GPU видны в `nvidia-smi`;
- `nvidia-smi topo -m` показывает NVLink внутри пар GPU 0/1 и GPU 2/3;
- NVMe-диск для swap установлен в слот с ожидаемой PCIe-пропускной способностью.

## 3. Сборка ML-окружения

Воспроизводимая установка Python, CUDA, PyTorch и vLLM описана в [world_build.md](./world_build.md). Основной вариант для NeuralTower:

- CUDA 12.8;
- Python 3.12;
- PyTorch с `cu128`;
- 1Cat-vLLM как основной путь для V100;
- официальный vLLM `0.18.x` как запасной путь.

Официальный vLLM 0.20+ не используется как базовый вариант для V100, потому что поддержка `sm_70` в нем не считается рабочей для этой платформы.

## 4. Запуск vLLM

Параметры инференса и объяснение TP/PP вынесены в [vllm_optimization.md](./vllm_optimization.md). Для топологии NeuralTower базовая схема распределения:

- `CUDA_VISIBLE_DEVICES=0,1,2,3`;
- `--tensor-parallel-size 2`;
- `--pipeline-parallel-size 2`;
- TP-группы остаются внутри NVLink-пар;
- межмезонинный обмен ограничивается передачей активаций между PP-стадиями.

Такой порядок снижает постоянный AllReduce-трафик через межмезонинный PCIe по сравнению с TP=4.

## 5. Мониторинг и диагностика

До длительных запусков моделей включите мониторинг из [scripts](./scripts):

```bash
cd Software/Linux/scripts
docker-compose up -d
./monitor.sh
```

Минимальный набор наблюдаемых параметров:

- температура GPU;
- температура HBM2;
- мощность каждого GPU;
- ширина и скорость PCIe-линков;
- температура жидкости;
- состояние помп;
- обороты вентиляторов;
- температура VRM.

Аппаратный тест V100 запускается из [../../Diagnosis/V100-SXM2-32G](../../Diagnosis/V100-SXM2-32G) до первого длительного инференса.

## 6. Последовательность первого запуска

1. Загрузить ОС без ML-нагрузки и проверить `dmesg` на ошибки PCIe/NVIDIA.
2. Проверить `nvidia-smi` и `nvidia-smi topo -m`.
3. Запустить DCGM-диагностику.
4. Включить мониторинг без нагрузки.
5. Запустить короткую CUDA-нагрузку на один GPU.
6. Повторить нагрузку на одну NVLink-пару.
7. Повторить нагрузку на все четыре GPU.
8. Запустить короткий vLLM-тест с малой моделью.
9. После тепловой стабилизации переходить к длинным контекстам и 70B-сценариям.
